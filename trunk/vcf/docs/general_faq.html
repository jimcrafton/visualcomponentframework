Q Why are you writing another framework ? Aren't there enough out there ?
A Please read this
Q Do I have to use CVS ?
A Yes and no (how that for a clear answer !). If you want to contribute to the VCF then VCS is the only way to check your work back in to the source tree. If you want an easy way to get the very latest code, and you don't want to wait till I put the next tgz file up, then CVS is also the way to go. However if you don't care about any of that then you can easily get the code by downloading the latest tgz file of the source tree.
Q Do I have to use SSH ? Why ?
A Yes and no. If you want write access to the CVS tree to check in code you have worked on and participate in the VCF's development, then yes you absolutely have to use SSH. This link explains SourceForge's policy about SSH as well as information about it and how it works.
Q What do I need to build ?
A Please read this, it will explain building the VCF.
Q Can I just get the binaries ?
A Currently I don not post just the binaries. The VCFInstaller.exe has options to install just the binaries if you like. If there is enough interest then I may change this.
Q Why on earth is the VCFInstaller.XXX.exe so big ?
A Because it has the whole enchilada ! It contains the following:
  • All the VCF source code for the core libraries, tests, and extra libraries, as well as any required 3rd party library code.
  • All the VCF binaries in both release and debug build. This include .lib files as well as .exe, and .dll files. Libraries for linking statically or dynamically are included.
  • All the source code for the current VC++ add-ins, as well as prebuilt binaries for these
  • The current html help generated documentation, both of the source code and a more general set of documentation. These can be integrated with MSDN if you have it installed.
  • cvs and ssh binaries and support files

Once you have installed this you are ready to start developing with the VCF. You can find out more about doing so here.
Q What systems does this run on ?
A Currently it runs on Win32 desktop platforms from Windows 98 (though Windows 95 should work, I just haven't tested it myself), Windows Me, Windows NT4.0 sp4 or better, Windows 2000, and Windows XP.

As of March 23, 2002, the FoundationKit compiles and runs (though not all peers are fully functional or implemented) on linux and Solaris 2.8.
Q What compilers does this build with ?
A For Win32 I use MS VC++ 6.0. There has been a great deal of interest to get it to work on Borland C++ compiler but so far no dice (as of March 23, 2002).

The FoundationKit will compile and build with GCC 2.95. I don't know about the GCC 3.x versions.
Q Can I use the VCF as a static library ?
A Yes. The naming convention is the library name (such as FoundationKit) and if it is a static library it is followed by and "s". If the library is a debug build it is followed by a "d". So the possible combinations for the FoundationKit (on Win32) .lib files are:
  • FoundationKit_d.lib (debug, DLL version )
  • FoundationKit.lib (release, DLL version )
  • FoundationKit_sd.lib (debug, Static lib version )
  • FoundationKit_s.lib (release static, Static lib version )
Q What is the memory footprint of the VCF ?
A In release mode (as of March 23, 2002)
  • a simple Console app (ala Hello World) takes 1.144 Mb 
  • a simple GUI Hello World takes 3.146 Mb

The size issue I believe is partially related to the large number of classes involved, many of which need to be exported. Several of the developers on the project have talked about this as well and we are going to be addressing this soon, by breaking up the modules into smaller units.
Q If I change the directory of the VCF (not of cvs/ssh), do I need to uninstall and reinstall VCF ?
A No. You do need to update the following environment variables:
  • VCF_INCLUDE
  • VCF_BIN
  • VCF_LIB
Q I am downloading all the files using cvs at the moment. Do I need to download VCFInstaller.exe as well to try it out ?
A No. The only advantage of the VCFInstaller is that it sets up everything for you, such as environment variables, binaries, etc. Otherwise you have to do this yourself, in which case you should follow this link for installation and this for building.
Q I downloaded all the files via cvs, and built the ApplicationKit_d.dll, FoundationKit_d.dll, and GraphicsKit_d.dll. Where do I go from here?

You can also get started by trying to create a simple application using the VC++ App Wizard that comes with the code (or VCF Installer). See here for more instructions.
Q What is the reasoning behind having RTTI for all the classes? Is that really needed? Isn't there some drawback to RTTI either larger memory footprint or slower execution or both?
A Component based development, basically. There are a number of things that need to happen dynamically, and for that you need RTTI. In other words you need dynamic discovery at runtime, and I use the  dynamic_cast<>  function to help with this since it already built in to the C++ system. 

I have tried to compile the VCF without RTTI (turning off support for the C++ language, and the additions of my own), and it seemed to make very little difference in memory foot print. So I think, but I'm not sure, that the size is due to the number of classes in the DLL. 

The speed issue is bogus in my opinion. I have done tests in release testing the speed of dynamic casts with reinterpret casts
These times were obtained using classes that were relatively deep in the hierarchy  (5 to 6 levels deep) and were obtained using the QueryPerformanceFrequency/QueryPerformanceCounter API, on a Windows 2000 box, Pentium 3 1Ghz CPU, 256 Mb RAMl, with VC++ 6 sp3.

The times are in seconds and represent the time it take to perform a dynamic_cast vs. a reinterpret_cast in a release build.

average reinterpret_cast time: 0.000002234921
average dynamic_cast time: 0.000005028572

that's pretty small
Q I just built a statically linked VCF hello world exe. It was 764K. I built the basically the same MFC app statically linking MFC. It was 252K. What gives ?
A Well partly the number of classes in VCF. The amount of functionality that sits inside of the ApplicationKit is much greater than what comes with MFC. However there are a number of classes that you may not need, which is why we are looking at ways to break it down and reduce this size.
Q Does VCF work under VC7 ?
A As of March 23, 2002, it does not. There are issues with with exporting templates that need to be changed. This was pointed out by another developer who had downloaded the framework and attempted to compile it on VC7. Since I do not currently have VC7 there is not much I can do. Anyone out there who does is welcome to help out on this !

Q What kind of license does the VCF use ?  I notice that many projects here on SourceForge use a GPL license, is this true of the VCF ?
A The VCF uses a BSD style license. In other words you are granted FULL rights to do whatever you would like with the code. You do NOT have to submit changes back, you do NOT have to make your source that uses the VCF available, so it is completely appropriate for commercial, proprietary development. I am more concerned with people using the VCF than I am trying to force people to do this, that, or the other with their code. However, it would be nice, though it is NOT required, if you gave credit where credit is due, so I would certainly appreciate being mentioned somewhere noticeable if you do decide to use the VCF.
Q Are you doing an open source project that will eventually provide the same IDE on LINUX, MS Windows, and Mac's as well? I didn't see anything about price when I was browsing your web page. How do you get paid?
A Yes the goals of the VCF are to run on Win32 (and I suppose Win64 at some point), linux, and Mac OSX. In addition the VCF Builderk, which is the GUI "IDE" for creating applications using the VCF, will also run on these platforms since it is written entirely in the VCF.

Sadly, I do not get paid for this (though wouldn't it be wonderful if someday that could happen...). I am doing this because I believe there is a need for something like this in C++, and I have yet to see anything else out there that comes close to the breadth of what the VCF can/will accomplish (with the exception of Qt but I can't afford $2,000 or more to get a toolkit that runs on multiple platform).
