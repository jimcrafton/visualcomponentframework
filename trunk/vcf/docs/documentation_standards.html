Documentation Standards

Introduction

This will explain how to write documentation topic for the VCF. This is considered separate from properly commenting code for the code generated documentation.

File Naming Standards

All topics will be in standard html, and will end with the ".html" (please don't end them in ".htm") file extension.

All topics will have a file name that is the same as the topic title, in lowercase, and with spaces substituted by underscore characters ("_" ). Topics that are considered too long may be trimmed in the file name. For example a topic titled "My Really Cool Controls" would be named "my_really_cool_controls.html".

All documentation should be stored in the vcf/docs directory - no subdirectories please.

Topic Skeletons

Alls topics will be made up of several subsections which will be labeled and named in the html with standard names. There two main topic types which determine which sections are used: the simplest type is a simple topic, which contains an Introduction, and zero or more sub topics (the exact formats to use will be discussed shortly). The second topic type is the more complex documentation topic, and contains two main sections: a Functional Specification, and a Usage section, as well as several sub sections under the two main sections. 

Simple Topics

Simple topics must have an Introduction and it should be labeled "Introduction" and use the H1 style. Each subtopic below this should use the H3 style. 

Documentation Topic

Each Topic will be made up of a Functional Specification section and a Usage section. The Functional Specification will be labeled "Functional Specification" and use the H1 style, and it will have an HTML name called "FunctionalSpecification". It will have four sub sections, each labeled "Introduction",  "Interfaces", "Classes", and "Detailed Description". All of these four sub sections will have HTML names "FS_Introduction", "FS_Interfaces", "FS_Classes", and "FS_Description", respectively. Each should use the H3 style. The Introduction should have a brief paragraph or two explaining the general intent or functionality of the topic. The Interfaces subsection should list all the interface classes used by the topic (there may be none). The Classes subsection should list all the classes used by the topic (there may be none). The Detailed Description should go into more detail and nitty-gritty of explaining how things work.

The Usage section will be labeled "Usage" and use the H1 style, and it will have an HTML name of "Usage". It will have four subsections, each labeled "Introduction", "Basic Usage", "Advanced Usage", and "Examples". All of these four sub sections will have HTML names "U_Introduction", "U_Basic", "U_Advanced", "U_Examples", respectively. Each should use the H3 style. The Introduction should have a brief paragraph or two explaining the usage at a high level. The Basic Usage should describe the most common or canonical usage. The Advanced Usage should describe how to use the topic in more advanced and complex ways, if applicable. The Examples should have some code sample further explaining how to use the topic.

There are template files which exist to serve as a starting point when creating a new topic.
